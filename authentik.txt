=== 邀请注册： ===
https://www.youtube.com/watch?v=mGOTpRfulfQ 包含从主页注册、和发送邀请链接（create users as inactive可以通过邮件确认变为active）
有两种方式，一种是从主页点击注册按钮跳转至注册（default authentication flow - Identification Stage
 - 配置Flow settings - Enrollment Flow），邮箱验证（create users as inactive），验证通过后才能登录（变为active用户）；另一种是发送邀请链接，填写邮箱但不用验证，直接登录（添加user-login stage）。
我的版本把这俩合并了，既要邀请链接，也要邮箱验证。

https://docs.goauthentik.io/users-sources/user/invitations/
https://github.com/goauthentik/authentik/issues/9719#issuecomment-3064702888

主页添加注册按钮，跳转至邀请码输入：
1. 登陆界面允许跳转到注册：
编辑Stages - default-authentication-identification - Flow settings，设置Enrollment flow
配置已登录用户直接跳转到主页：Flow Overview - Edit - Behavior settings - Deniced action - CONTINUE
2. 自定义邀请码：
	1. Prompts新增一个输入邀请码的field，Field Key设为invitation-code
	2. Policies新增Expression Policy
	''if request.context.get('prompt_data',{}).get('invitation-code','') == 'mone':''
		''return True''
	''ak_message("邀请码错误！")''
	''return False''
	3. 新增一个 Prompt stage，添加邀请码field和上述policy
	4. 新增一个 Enrollment Flow (Authentication: require no authentication)，添加上述stage，
3. 再添加一个 redirect stage 跳转到真正注册用的flow（创建见下）

创建真实注册入口（已创建，存为 my register Flow (inactive until email confirmation) with default access to emby-dd）
1. 新建 Flow: 
	Title：Welcome to authentik!
	Designation: Enrollment
	Authentication: Require being redirected from another flow
(old: 下载 https://goauthentik.io/blueprints/example/flows-enrollment-2-stage.yaml , 在Flow里import)
2. 添加Prompt Stage 填写注册信息：
	添加一个 Prompt Stage，选择要填写的Field （要等一会才能加载出来）（用户名 - 密码 - 重复密码 - 邮箱 - 昵称）
	（编辑可填写的Fields：Flows and Stages - Prompts，可以在这里改变他们的显示顺序）
	Policy可以先清空

	添加密码复杂度要求：
	Custom - Policies创建Password Policy
	编辑刚才的Prompt Stage - Edit Stage， 选择刚添加的 Policy

	添加邮箱唯一性要求：
	https://docs.goauthentik.io/customize/policies/expression/unique_email/
	在Customization - Policies里新建一个Expression Policy
		''# Ensure this matches the *Field Key* value of the prompt''
		''field_name = "email"''
		''email = request.context["prompt_data"][field_name]''
		''pending_user = request.context.get("pending_user")''
		
		''from authentik.core.models import User''
		''query = User.objects.filter(email__iexact=email)''
		''if pending_user:''
			''query = query.exclude(pk=pending_user.pk)''
		
		''if query.exists():''
			''ak_message("Email address in use")''
			''return False''
		
		''return True''
	编辑刚才的Prompt Stage - Edit Stage， 选择刚添加的 Policy
	
3. 添加User-write stage 写入新增用户
	Create new users when required
	如果勾选Create users as inactive，则需要等待邮箱验证通过才能登陆，否则可以直接登录
	User type: Internal
	User path template: users/emby-dd
	Group: emby-dd autologin visible  （新注册用户自动添加到这个group https://next.goauthentik.io/users-sources/user/invitations/#automatic-group-assignment%EF%BC%89
4. 添加Email stage 验证邮箱
	添加邮箱认证：
	1. 在.env中添加（只开SSL，关闭TLS；__FROM好像也不能乱改）
		''# SMTP Host Emails are sent to''
		''AUTHENTIK_EMAIL__HOST=smtp.qq.com''
		''AUTHENTIK_EMAIL__PORT=465''
		''# Optionally authenticate (don't add quotation marks to your password)''
		''AUTHENTIK_EMAIL__USERNAME=850462618@qq.com''
		''AUTHENTIK_EMAIL__PASSWORD=jiybjuqdepmfbdij''
		''# Use StartTLS''
		''AUTHENTIK_EMAIL__USE_TLS=false''
		''# Use SSL''
		''AUTHENTIK_EMAIL__USE_SSL=true''
		''AUTHENTIK_EMAIL__TIMEOUT=10''
		''# Email address authentik will send from, should have a correct @domain''
		''AUTHENTIK_EMAIL__FROM=850462618@qq.com''
	2. 然后重新运行 docker compose up -d
		测试SMTP配置：
		ak test_email <to_address> [-S <stage_name>]
	3. 添加Email stage，使用global connection settings（不知道为什么单独的settings一直报SSL错，还是用全局的吧）
		Activate pending user on success
		Use global settings
		Subject: 设置邮件标题 
		Template: Account Confirmation
	邮件里验证网址的host会自动修改为当前使用的host，不用专门配置
5. (可选) 允许直接跳转到登录后页面：添加user-login stage（如果需要邮箱验证就不加这一步！）

=== Provider： ===
配置一个认证服务器。接入的http应用会通过访问provider，管provider要鉴权信息。

添加OAuth / OIDC Provider:
参考 [[webdav:Alist]]
Authorization Flow： implicit 和 explicit 的区别是后者多点一步确认

Proxy Provider：
三种分类目前的理解：
	Proxy：把自建服务用authentik (outpost)自动 包一层反代。authentik (outpost) 作为反代服务，访问External Host就是authentik (outpost) 的反代服务器，通过认证之后，跳转到Internal Host自建服务。（当然authentik和自建服务不用在同一台主机上）参考：https://integrations.goauthentik.io/media/tautulli/#authentik-configuration
	Forward Auth：
		自己手动给自建服务包一层反代（如nginx、NPM、traefik），一个自建服务对应一个反代，现在反代就看做自建服务本身，手动配置与 authentik (outpost) 的交互逻辑，用户访问External Host就是访问自己的反代服务器。
		这样的好处就是自己反代的可控性更高，比如可以把鉴权入口放在一个单独的path，或者完全可以没有内部服务，而是直接302到实际服务，但前提是实际服务需要支持带token的访问方式，否则没法把鉴好的权传过去。

=== Outpost: ===
https://docs.goauthentik.io/add-secure-apps/outposts/
https://next.goauthentik.io/add-secure-apps/outposts/embedded/

把provider的活抽离出来，负责与自建服务的api进行交互鉴权，一般是跟自建服务部署在同一主机上，方便自建服务跟outpost指定的端口进行交互。
一个自建服务对应一个outpost。
outpost则通过api与authentik交互。
默认会有一个authentik embeded outpost部署在authentik主机上。

如果自建服务运行在本机，可以使用自动配置的embedded outpost：
	编辑/新建outpost——integration选Local Docker connection即可
	由authentik管理，会自动连接本地docker，然后自动创建一个outpost容器。
	如果是Proxy类型的，端口也是9000、9443，会跟authentik冲突，所以要么用默认的authentik embeded outpost，要么下面的手动创建。
	
如果自建服务在别的主机，考虑手动创建outpost，再配置outpost连接authentik
/如果自建服务在docker里，也可以考虑手动创建outpost，让outpost与自建服务同时开关
	https://docs.goauthentik.io/add-secure-apps/outposts/manual-deploy-docker-compose/
	参考：https://www.bilibili.com/video/BV1Mb421a7fF  4:23
	outpost用的也是9000和9443，所以如果和authentik在同一主机上，需要修改端口

高级设置 outpost所在容器的一些设置：参考：https://www.bilibili.com/video/BV12z421C7hU/ 11:45
https://next.goauthentik.io/add-secure-apps/outposts/embedded/
	embedded outpost修改：
	authentik_host: http://authentik.barryzzj.top:23354/ #outpost如何访问authentik。#如果是embedded outpost，经测试无论如何都能找到，这一项其实充当authentik_host_browser的作用。此时authentik_host_browser不起作用。

	手动创建outpost修改：
	authentik_host: http://authentik.barryzzj.top:23354/ #outpost如何访问authentik。如果在同一主机，%E5%8F%AF%E4%BB%A5%E8%AE%BE%E7%BD%AE%E4%B8%BAhttp://authentik-server-1:9000/ (来自docker ps)。
	docker_network: authentik_default  #(来自docker network ls) # non-embedded only
	authentik_host_browser: http://authentik.barryzzj.top:23354/ #引导用户浏览器如何访问authentik
